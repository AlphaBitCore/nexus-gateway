# Trace ID / Request ID Propagation Architecture

> **Tier 1 architecture doc.** Read this before adding a new cross-service event, debugging a multi-service flow, changing middleware that touches `x-nexus-request-id`, or extending the audit envelope. The MQ envelope conventions live in `mq-architecture.md`; the audit consumer that stores these fields lives in `audit-pipeline-architecture.md`.

Cross-service correlation in Nexus uses **two** identifiers, deliberately distinct:

- **`request_id`** — Nexus's own request-correlation id. Created at the edge that first sees the request (the AI Gateway for `/v1/*` calls; the Control Plane for admin API calls; the agent for an intercepted application request).
- **`trace_id`** — OpenTelemetry trace id. Created or extracted at the same edge; propagated via W3C `traceparent` headers when calling outward.

Both flow end-to-end. UI surfaces and audit queries join on `request_id`; OTel exporters and tracing UIs join on `trace_id`.

---

## 1. Identifier semantics

### `request_id`

- UUID v4 random (`uuid.New()` in `packages/nexus-hub/internal/handler/middleware.go::NexusRequestID`).
- Header: `x-nexus-request-id` (lowercase; case-insensitive on read).
- Generated by the **first Nexus service** that handles the request. The middleware **reuses** an inbound `x-nexus-request-id` header if present — this lets clients pre-correlate (the response will still carry the same id back), and lets internal service-to-service calls inherit the upstream id without a new one being minted per hop.
- Stamped on every audit row, traffic event, log line, and metric exemplar associated with the request.
- Persisted across MQ envelopes and HTTP fallback paths.

### `trace_id`

- W3C trace context (16-byte trace id + 8-byte span id).
- Header: `traceparent` (and `tracestate` for vendor extensions).
- Extracted from inbound `traceparent` if present and valid; otherwise generated.
- Propagated outbound on every HTTP call to providers and to internal services.
- Used by OTel exporters and tracing back-ends. We do NOT use `trace_id` as the primary key for analytics queries — that's `request_id`'s job.

`request_id` and `trace_id` are recorded together. UI surfaces show both; ops typically searches by `request_id` first.

## 2. Edge generation

Three edges generate ids:

| Edge | Service | Trigger |
|---|---|---|
| `/v1/*` request | AI Gateway | First middleware in the chain. |
| Admin API request | Control Plane | First middleware in the chain. |
| Application-intercepted request | Agent (or Compliance Proxy) | On CONNECT / on flow accept. |

The middleware that does this is `packages/nexus-hub/internal/handler/middleware.go` for Hub-side requests and the equivalent in each service. The header name is uniform: `X-Nexus-Request-ID`. **Always** call the shared middleware constructor — don't roll your own.

## 3. HTTP propagation

When a Nexus service calls another Nexus service over HTTP:

- `X-Nexus-Request-ID` is forwarded verbatim.
- `traceparent` is updated per W3C semantics (new span id, same trace id).

When a Nexus service calls an external provider:

- `X-Nexus-Request-ID` is NOT forwarded (providers don't know what to do with it).
- `traceparent` IS forwarded (some providers honour it; harmless if they don't).

The `packages/shared/transport/http/` package handles both automatically; service code never sets these headers manually on the outbound side. The inbound request id is threaded into the context via `nexushttp.WithRequestID(ctx, id)` and the outbound client reads it back out from the context — there is no auto-injection of `traceparent` at the Nexus layer; OTel's HTTP middleware (`packages/shared/core/telemetry/httptrace.go`) is what adds `traceparent` to outbound calls.

## 4. MQ propagation

The traffic event envelope (`packages/shared/transport/mq/messages.go::TrafficEventMessage`) carries the ids on its top-level fields:

```jsonc
{
  "id":                "01HXYZ...",        // event id (dedup key)
  "source":            "ai-gateway",
  "traceId":           "0af7651916cd43dd8448eb211c80319c",
  "externalRequestId": "01HXYZ-REQ-...",
  // …timing, identity, traffic detail…
  "nexusRequestId":    "01HXYZ-REQ-..."     // appears on the agent-audit message variant
}
```

Consumers preserve these fields when forwarding the event downstream. NATS subject (`nexus.event.ai-traffic`) is enough for routing; the message body is the only correlation surface in current code (no parallel NATS-header copy).

## 5. Audit storage

- `traffic_event` carries a `nexus_request_id` column and a `trace_id` column; both are indexed. The TLS-bumped `X-Nexus-Trace-ID` header is extracted onto `traffic_event.trace_id` for compliance-proxy/agent flows where the upstream caller propagated a trace id.
- `AdminAuditLog` carries a `nexusRequestId` column (indexed); it does NOT carry an OTel `trace_id` column. Admin-action correlation is by request id alone.
- There is no `agent_audit` table — agent traffic rows land in `traffic_event` with `trafficSource = AGENT`.

A single `nexusRequestId` lookup retrieves the traffic-event row plus any admin-audit row stamped by the same admin request. The CP "Request Timeline" UI joins on `nexus_request_id` / `nexusRequestId`.

## 6. Agent ↔ server stitching

When an agent intercepts an application request:

1. Agent generates `request_id` and `trace_id` locally (the agent IS the edge for endpoint-intercepted traffic).
2. Agent runs the local pipeline (hooks, etc.), stamps both ids on its local SQLCipher audit queue.
3. When forwarding to upstream, agent uses HTTP client behaviour from §3.
4. On audit upload, agent emits with the same `request_id` and `trace_id`.

For requests where the agent's intercepted application **itself** calls Nexus AI Gateway (a redundant path; not the typical setup), both ends will generate request_ids — agent-edge id + AI-Gateway-edge id. The agent-edge id is what surfaces in the unified timeline; the AI-Gateway-edge id is stored separately for cross-reference.

## 7. Service-internal trace_id (not request_id)

Within a single service, span correlation uses OTel's trace_id + span hierarchy. Spans don't need their own request_id — they inherit from the request middleware's stamping.

Background jobs (Hub scheduler, MQ consumers) generate a fresh `request_id` per job execution. The `trace_id` for a job is also fresh (no inbound context). Audit rows emitted by jobs carry this synthetic `request_id`, prefixed with `job-` so ops know it didn't originate from a user request.

## 8. Null-tolerance for analytics

Rows produced outside an incoming-request context (background jobs, sweepers, retroactive imports) may carry a synthetic `request_id` or a NULL. Analytics queries should tolerate NULL `request_id` and surface a "no request id" badge for those rows.

## 9. Common debugging recipes

- "Why did this request 429?" — Search `traffic_event WHERE request_id = ...` → look at `error_class` + `hook_decision`.
- "What triggered this alert?" — Alert rows carry sample `request_id`s → fan out from there.
- "Where did this agent traffic come from?" — `traffic_event WHERE nexus_request_id = ... AND traffic_source = 'AGENT'` → look at `source_process` / `source_process_bundle`.
- "Did the same request hit cache?" — `traffic_event.cache_hit` is per-row; group `request_id` if a cache lookup fan-out occurred.

## 10. Sources

- `packages/nexus-hub/internal/handler/middleware.go` — `NexusRequestID` middleware (UUID v4 + inbound-header reuse).
- The equivalent middleware in each service (CP, AI GW, Compliance Proxy, Agent).
- `packages/shared/core/telemetry/` — OTel setup; `traceparent` extraction; HTTP-middleware that injects `traceparent` outbound.
- `packages/shared/transport/http/` — `nexushttp.WithRequestID` context propagation for `x-nexus-request-id`.
- `packages/nexus-hub/internal/traffic/ingest/audit/agent_audit.go` — agent audit upload preserves both ids.
- `packages/shared/transport/mq/messages.go` — MQ envelope (`TrafficEventMessage`) top-level `traceId` / `nexusRequestId` fields.
