# E31: Nexus Response Markers — Requirements

**Epic:** E31 (AI Gateway client simulator and traffic visibility)

## Functional Requirements

**FR-1 (Must)** — Every Nexus service that intercepts and processes a request MUST emit identifying markers on the response:
- `x-nexus-via` chain entry
- `x-nexus-trace-id` (cross-service correlation)
- Per-service `x-nexus-<svc>-*` set (request-id, mode, hook outcome, plus service-specific dimensions)

**FR-2 (Must)** — Markers MUST appear on success paths AND reject paths.

**FR-3 (Must)** — Markers MUST NOT appear on forwarding/tunnel paths (where Nexus does not terminate TLS or process the request).

**FR-4 (Should)** — Markers SHOULD be readable from browser-side JavaScript via `fetch()` — requires `Access-Control-Expose-Headers` containing the marker names.

**FR-5 (Should)** — The chain SHOULD reflect request flow order so a reader can determine which Nexus services processed the request.

**FR-6 (Should)** — Hook outcomes SHOULD be structured (`none` / `passed:<hook>` / `transformed:<hook>` / `rejected:<hook>:<reason>`) so they are machine-parseable.

## Non-Functional Requirements

**NFR-1 (Must)** — Marker emission MUST add no measurable latency (header sets are O(1)).

**NFR-2 (Must)** — Marker emission MUST preserve upstream framing (no chunked-encoding bugs, no double-Content-Length).

**NFR-3 (Must)** — Hook reject reasons MUST be sanitized to `[a-z0-9-]+` to prevent header-injection and log-poisoning.

**NFR-4 (Should)** — Header naming convention is stable for the pre-GA development window; long-term contract revisited at GA.

## User Roles & Personas

- **Operator** — debugs traffic via DevTools or curl; reads markers to identify Nexus involvement and hook outcomes without consulting audit logs.
- **Customer support engineer** — uses `x-nexus-trace-id` to correlate a specific request to its audit record.
- **Browser-side application developer** — reads markers via `fetch()` for client-side observability dashboards or automated assertions.

## Constraints & Assumptions

- Pre-GA: no backward-compatibility constraints on header names; naming may be revised before GA.
- Tunnel and non-MITM paths are observable only via Nexus Hub flow inventory and audit log (out of scope for this feature).
- Streaming responses lose response-side dimensions (cache hit/miss, upstream latency, response hook outcome, final usage); these are captured in the audit event instead.
- `x-nexus-trace-id` is generated by the most upstream Nexus service and propagated downstream via the internal request context; it does not depend on any external tracing infrastructure.

## Glossary

| Term | Definition |
|---|---|
| **MITM** | Man-in-the-middle; a Nexus service terminates TLS to inspect request/response. |
| **Hook** | A compliance pipeline component (e.g. pii-redact, jwt-strip) that runs inline on the request or response body. |
| **Via chain** | Comma-separated list of Nexus service names that processed a request, in request flow order. |
| **Tunnel path** | A forwarding path where Nexus relays bytes without terminating TLS; no marker emission. |
| **Reject path** | A path where a Nexus service halted the request (hook rejection, policy block, quota exceeded). |
