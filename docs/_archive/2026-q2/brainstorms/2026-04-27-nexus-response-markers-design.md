# Nexus Response Markers — Design

**Date:** 2026-04-27
**Status:** Draft (post-brainstorming, pre-implementation)
**Owner:** nexus
**Related:** E31 (AI Gateway client simulator and traffic visibility)

## 1. Background and Problem Statement

Today, only the AI Gateway emits response headers that identify Nexus involvement
(`x-nexus-request-id`, `x-nexus-cache-status`, `x-nexus-provider`, etc.). The
Compliance-Proxy success path forwards upstream headers verbatim, and the Agent's
MITM relay byte-streams the response opaquely. As a result, an operator looking at
a response in browser DevTools or a `curl -i` cannot answer:

- Was this request actually intercepted by Nexus, and by which services?
- Did any Nexus hooks run? Were they passed, transformed, or rejected?
- Did the response come from cache?
- Which provider/model/routing rule served it?

The existing AI-Gateway-only headers also use inconsistent casing
(`x-nexus-request-id` Pascal vs. `x-nexus-cache-status` lowercase) and have no
namespace strategy for when multiple Nexus services participate in a single flow.

A separate question surfaced during discussion — per-model parameter constraints
(e.g. GPT-5.1 rejecting `temperature=0.7`) — was explicitly **closed without
action**: Nexus will not validate or coerce parameters; upstream errors pass
through unchanged. See §11 D-1 for the one related loose end (the existing
`max_tokens` rename for reasoning models, kept with a transparency header).

## 2. Goals

1. Every Nexus service that **actually processes** (intercepts, MITM-decrypts,
   runs hooks on) a request emits a uniform set of response headers identifying
   itself and what it did.
2. The header surface is consumable by:
   - Browser DevTools Network tab (no JS needed)
   - `curl -i` / shell tooling
   - Browser-side JS (via CORS expose)
3. A reader can determine, from the response alone:
   - Which Nexus services were on the path (`x-nexus-via`)
   - Per service: whether hooks ran and which ones, the operating mode, the
     interception rule that matched (where applicable)
   - For AI Gateway specifically: provider, model, routing rule, cache status,
     quota state, latency, retry count
4. Header naming is consistent (`x-nexus-<service>-<dim>`, all lowercase).

## 3. Non-Goals

- **Per-model parameter validation or coercion.** Out of scope. Upstream errors
  pass through. (Topic A discussion concluded "let upstream errors surface as-is.")
- **Stamping markers on forwarding/CONNECT-tunnel flows.** Physically impossible
  without TLS termination; not addressed via side channels. Such flows are
  observable only via Nexus Hub flow inventory / audit log.
- **SSE trailer events or HTTP/1.1 trailers** for late-bound data. Streaming
  responses get only request-side determinable markers; the rest goes to audit
  log, queryable by `x-nexus-trace-id`.
- **Backward compatibility with existing header names.** Per project pre-GA policy
  ("no backcompat, no defer"), old names are removed in the same change set.

## 4. Header Contract

### 4.1 Cross-service headers (set by every hop that processes the request)

| Header | Semantics | Producer behavior |
|---|---|---|
| `x-nexus-via` | Comma-separated chain in **request flow order** (`agent, compliance-proxy, ai-gateway`) | Each hop on the response path: if the header is absent on the incoming response, **initialize** with own service name; otherwise **prepend** own service name to the existing value. The deepest Nexus hop (whichever service is the last Nexus code in the request path — could be AI Gateway, CP-without-AIGW, or Agent-direct-to-upstream) naturally initializes. Absence of this header on the final response indicates no Nexus service processed the request. |
| `x-nexus-trace-id` | Cross-service correlation UUID | Generated by the most upstream Nexus service to see the request (typically Agent). Propagated unchanged through every hop. |

The pre-existing request-side header `x-nexus-request-id` (used as a cross-service
txID) is **removed**. Cross-service correlation uses `x-nexus-trace-id`
exclusively. Each service has its own per-service request ID (see below).

### 4.2 AI Gateway (`x-nexus-aigw-*`)

All currently-emitted AI Gateway headers are renamed under the `aigw` prefix.

| New name | Replaces | Semantics |
|---|---|---|
| `x-nexus-aigw-request-id` | `x-nexus-request-id` (response-side) | AI Gateway audit `Record.RequestID`. |
| `x-nexus-aigw-mode` | (new) | Always `proxied` for AI Gateway; reserved for future modes. |
| `x-nexus-aigw-hook` | (new) | Hook outcome — see §4.5. |
| `x-nexus-aigw-provider` | `x-nexus-provider` | Resolved provider slug. |
| `x-nexus-aigw-model` | `x-nexus-model` | Customer-facing model code (the value the caller sent in `{"model": "..."}`). |
| `x-nexus-routed-provider` | `x-nexus-routed-provider` | Set when smart-routing substituted the provider. |
| `x-nexus-routed-model` | `x-nexus-routed-model` | Set when smart-routing substituted the model. |
| `x-nexus-aigw-routing-rule` | `x-nexus-routing-rule` | Name of the routing rule applied. |
| `x-nexus-cache` | `x-nexus-cache-status` | `hit` / `miss` / `bypass` / `disabled`. Omitted on streaming responses. |
| `x-nexus-aigw-stream` | `x-nexus-stream-live` | `true` on streaming responses. |
| `x-nexus-aigw-latency-ms` | `x-nexus-latency-ms` | Wall-clock ms. Omitted on streaming responses. |
| `x-nexus-aigw-retry-count` | `x-nexus-retry-count` | Set when retries occurred. |
| `x-nexus-quota-used` / `-limit` / `-period` / `-remaining` | `x-nexus-quota-*` | Existing semantics. Omitted on streaming responses. |
| `x-nexus-quota-downgrade` / `-quota-original-model` / `-quota-warning` | `x-nexus-quota-*` | Existing semantics. |
| `x-nexus-coerced` | (new) | Comma-separated list of `<from>→<to>` rewrites that the gateway applied to the request before sending upstream (currently only `max_tokens→max_completion_tokens` for `gpt-5*` / `o1+` reasoning models). Omitted when no rewrite occurred. See §11 D-1. |

Request-side AI-Gateway-specific headers are also renamed:
- `X-Nexus-Body-Format` → `x-nexus-aigw-body-format`
- `X-Nexus-No-Cache` → `x-nexus-aigw-no-cache`

### 4.3 Compliance-Proxy (`x-nexus-cp-*`), MITM path only

| Header | Semantics |
|---|---|
| `x-nexus-cp-request-id` | CP's per-request UUID (currently the `txID` it generates). |
| `x-nexus-cp-mode` | Always `mitm`. (Tunnel/forwarding flows do not enter the marker code path.) |
| `x-nexus-cp-hook` | Hook outcome — see §4.5. |
| `x-nexus-cp-domain-rule` | UUID of the matched interception-domains rule (per §11 D-3). |

Reject path: stops setting `X-Nexus-Hook-Decision` / `X-Nexus-Hook-Reason`. The
reject information is fully encoded in `x-nexus-cp-hook: rejected:<hook>:<reason>`.

### 4.4 Agent (`x-nexus-agent-*`), MITM path only

| Header | Semantics |
|---|---|
| `x-nexus-agent-flow-id` | Agent's internal flow UUID (correlates to local agent audit). |
| `x-nexus-agent-mode` | Always `mitm`. |
| `x-nexus-agent-hook` | Hook outcome — see §4.5. |
| `x-nexus-agent-domain-rule` | UUID of the matched interception-domains rule. |

Reject path: stops setting `X-Nexus-Hook-Decision` / `X-Nexus-Hook-Reason`,
replaced by `x-nexus-agent-hook: rejected:<hook>:<reason>`.

### 4.5 Hook outcome value format

Used identically for `x-nexus-aigw-hook`, `x-nexus-cp-hook`, `x-nexus-agent-hook`:

```
none                                # No hook in this service ran for this request
passed:<hook1>,<hook2>              # Hooks ran; all passed without modifying body
transformed:<hook1>,<hook2>         # Hooks ran; at least one modified the body
rejected:<hook>:<reason-slug>       # First hook to reject. Halts pipeline.
```

- `<hook*>` is the hook's stable slug (e.g. `pii-redact`, `jwt-strip`), not its
  display name. Lowercase, kebab.
- `<reason-slug>` is the hook's machine-readable reject reason code, not free
  text. Constrains characters to `[a-z0-9-]+` to prevent header-injection /
  log-poisoning.

## 5. Streaming Response Behavior (S1)

For SSE / streaming responses, headers must flush before the first byte; only
**request-side determinable** information is included.

**Present in streaming responses:**
- `x-nexus-via`
- `x-nexus-trace-id`
- All `*-request-id` / `*-flow-id`, `*-mode`, `*-domain-rule`
- `x-nexus-aigw-provider`, `-model`, `-routed-*`, `-routing-rule`
- Request-side hook results: `x-nexus-{aigw,cp,agent}-hook` (request pipeline
  completes before first byte today in all three services; verified during
  exploration).
- `x-nexus-aigw-stream: true`

**Omitted in streaming responses:**
- `x-nexus-cache` (no full-body cache hit semantics for streams)
- `x-nexus-aigw-latency-ms` (final latency unknown at flush time)
- `x-nexus-quota-used` / `-remaining` (final token count unknown)
- Response-side hook info (output PII redaction, etc.)

The omitted dimensions are still recorded in the `traffic_event` audit row and
are queryable via `x-nexus-trace-id`. Admin UI traffic detail page already links
by trace-id, so no new query API is required.

## 6. Reject Path

Hook rejects produce an `HTTP/1.1 403`-style response constructed directly by
the rejecting service. Marker behavior:
- `x-nexus-via` includes only services that were actually entered. If CP rejects,
  AI Gateway is not on the chain — `x-nexus-via: compliance-proxy`.
- `x-nexus-<svc>-hook: rejected:<hook>:<reason>` is the sole carrier of reject
  information. The legacy `X-Nexus-Hook-Decision` / `X-Nexus-Hook-Reason` headers
  are deleted.
- Reject responses include the same `Access-Control-Expose-Headers` configuration
  (§7) so browsers can read markers on 403 too.
- Response body envelope (the JSON / HTML reject page) is unchanged.

## 7. CORS Exposure

To make `x-nexus-*` readable from browser-side JavaScript via `fetch` / `XHR`,
each service's HTTP entry must emit (wildcards are not universally supported in
`Expose-Headers`; explicit list required):

```
Access-Control-Expose-Headers: x-nexus-via, x-nexus-trace-id,
  x-nexus-aigw-request-id, x-nexus-aigw-mode, x-nexus-aigw-hook,
  x-nexus-aigw-provider, x-nexus-aigw-model,
  x-nexus-routed-provider, x-nexus-routed-model,
  x-nexus-aigw-routing-rule, x-nexus-cache, x-nexus-aigw-stream,
  x-nexus-aigw-latency-ms, x-nexus-aigw-retry-count,
  x-nexus-quota-used, x-nexus-quota-limit,
  x-nexus-aigw-quota-period, x-nexus-aigw-quota-remaining,
  x-nexus-quota-downgrade, x-nexus-quota-original-model,
  x-nexus-quota-warning, x-nexus-coerced,
  x-nexus-cp-request-id, x-nexus-cp-mode, x-nexus-cp-hook,
  x-nexus-cp-domain-rule,
  x-nexus-agent-flow-id, x-nexus-agent-mode, x-nexus-agent-hook,
  x-nexus-agent-domain-rule
```

(The exact authoritative list lives in `packages/shared/traffic/expose_headers.go`
as a `var ExposeHeaders []string`. Each service builds its `Expose-Headers`
value from a slice import to avoid drift.)

Two emission patterns by response path:

**Success path (CP / Agent transparent proxy):** the upstream response may
already carry an `Access-Control-Expose-Headers` set by the origin. The
service-side helper `traffic.MergeExposeHeaders(h http.Header, names ...string)`
appends the Nexus marker names to any existing value (case-insensitive
deduplication, preserves upstream's entries).

**Synthetic responses (reject path; AI Gateway non-proxied responses):** there
is no upstream `Access-Control-Expose-Headers` to merge against. The service
**directly emits** the full Nexus marker list via
`traffic.SetExposeHeaders(h http.Header)`.

Implementation:
- **AI Gateway**: success and reject responses both go through
  `setResponseHeaders()`; uses `SetExposeHeaders` (AI Gateway is the origin from
  the client's perspective, no upstream CORS state). Wire up next to the
  existing CORS `Allow-Headers` config in
  `packages/ai-gateway/cmd/ai-gateway/main.go:824`.
- **Compliance-Proxy / Agent transparent success path**: `MergeExposeHeaders`
  after the upstream header copy.
- **Compliance-Proxy / Agent reject path**: `SetExposeHeaders` directly on the
  synthetic 403 response (no upstream to merge with).

## 8. Implementation Surface

### 8.1 Phase 0 — Prerequisite: hoist response handling to shared

Today the Agent's MITM relay byte-streams responses opaquely; it does not parse
response headers, so it cannot inject markers. Compliance-Proxy already has full
response parsing, header forwarding, and SSE accumulation in
`packages/compliance-proxy/internal/proxy/upstream.go` and `sse.go`.

Phase 0 hoists those capabilities into a new shared package
(`packages/shared/transport/responseio`, exact name TBD during implementation) and migrates
Compliance-Proxy onto it as a no-behavior-change refactor. Agent then depends on
the same package in subsequent phases. This unlocks not only response-header
injection (this spec) but also future response-side hooks in the agent.

### 8.2 Per-service write points

**AI Gateway** (`packages/ai-gateway/internal/handler/proxy.go`):
- `setResponseHeaders()` (~line 1049) — non-streaming + streaming common baseline.
- `handleStream()` (~line 1077) — flush only request-side markers; explicitly
  omit cache / latency / response-hook fields.
- Cache-hit fast-path (~line 377) — `x-nexus-cache: hit` + baseline.
- Quota block (~line 653) — rename `x-nexus-quota-*` → `x-nexus-aigw-quota-*`.
- Hook executor completion site — write `x-nexus-aigw-hook` via
  `formatHookOutcome()` helper.

**Compliance-Proxy** (`packages/compliance-proxy/internal/proxy/`):
- `upstream.go:141` — non-streaming success path. After upstream header copy,
  add `x-nexus-cp-*` and `PrependVia`.
- `sse.go:61` — streaming success path. Same logic, omit response-side hook info.
- `reject.go:124,178` — replace `X-Nexus-Hook-Decision/Reason` with
  `x-nexus-cp-hook: rejected:<hook>:<reason>` and `via`.
- `forward_handler.go` — at the interception-domains match site, store the
  matched rule UUID and the request-side hook outcome in the request context so
  that downstream write sites can read them.

**Agent** (`packages/agent/core/network/proxy/proxy.go`):
- MITMRelay response-injection step (added under existing `respInspector`
  parameter, depends on Phase 0 infra). Parse-mutate-write the response headers
  using `shared/responseio`; body bytes continue to stream through unchanged
  (preserves SSE / chunked semantics).
- Reject path (~line 538) — replace legacy hook headers with
  `x-nexus-agent-hook` + `via`.

### 8.3 `x-nexus-via` chain construction

Every hop **prepends** itself, so the final value reads in request flow order
(`agent, compliance-proxy, ai-gateway`). Helper `traffic.PrependVia(h, svc)`
lives in `packages/shared/traffic`, used by all three services.

## 9. Audit Recording

The `traffic_event` table already records the underlying fields (cache_status,
hook outcome, provider, model, etc.). **No new "headers snapshot" column** is
added; markers and audit records are populated from the same in-process state,
so they cannot diverge.

For dimensions intentionally omitted from streaming response headers (cache,
latency, response-hook, final usage), the audit row is the source of truth and
is reachable via `x-nexus-trace-id` on the Admin UI traffic detail page.

## 10. Testing Strategy

Per the project's mandatory workflow, every change ships with `go test -race
-count=1`-passing tests.

### 10.1 Unit tests

- `formatHookOutcome()` table-driven: empty hooks list, single pass, multiple
  pass, transformed, reject + reason slug containing edge characters (verifies
  slug validation rejects unsafe input).
- `PrependVia` / `MergeExposeHeaders` table-driven: empty header, existing
  single value, existing multi-value, duplicate suppression, case handling.
- AI Gateway `setResponseHeaders` table-driven: matrix of
  `{stream, cache_status, hook_outcome, route_substituted, quota_state}` × the
  expected emitted header set, with explicit assertions that streaming responses
  do **not** carry the omitted dimensions.

### 10.2 Integration tests

- AI Gateway `internal/handler/integration_test.go`: existing cases updated for
  renamed headers; new "streaming response omits cache/latency" case.
- Compliance-Proxy `internal/proxy/*_test.go`: MITM success, reject, and
  streaming, all asserting `x-nexus-via` and the full `x-nexus-cp-*` set.
- Agent `internal/proxy/*_test.go`: MITM success path asserts `x-nexus-agent-*`
  injection. CONNECT-tunnel path asserts the **absence** of any `x-nexus-*`
  marker (negative test, prevents regression of the "tunnel = no marker" rule).

### 10.3 End-to-end tests

A new e2e (under either CP or AI Gateway test suite, TBD) sends a request
through CP → AI Gateway and asserts:
- `x-nexus-via: compliance-proxy, ai-gateway`
- Both services' full marker sets are present
- `x-nexus-trace-id` is identical end-to-end

### 10.4 Front-end (optional)

Vitest for the AI Gateway Simulator page: with mocked responses bearing the new
markers, the UI surfaces a "processed by Nexus" panel. Out of scope for the core
spec but flagged for the implementation plan to consider.

## 11. Decisions Log

(All previously-open questions resolved during brainstorming on 2026-04-27.)

**D-1 — `applyOpenAIReasoningRewrites` (`max_tokens → max_completion_tokens`
for `gpt-5*` / `o1+` reasoning models)** — **Decided: keep the rewrite + emit
transparency header.**

Rationale: the rewrite is a **vendor rename** (same semantics, different field
name), not a value constraint, so it does not violate the "upstream errors
surface as-is" policy. Removing it would break callers using the pre-reasoning
OpenAI shape against `gpt-5*` / `o1+` for what is effectively a vendor-side
rename. To satisfy the "no silent coerce" principle, the AI Gateway emits
`x-nexus-coerced: max_tokens→max_completion_tokens` on any response where
the rewrite was applied. The header is extensible (comma-separated list) for
future renames in the same class. See §4.2 for the header definition.

**D-2 — Reject chain on `x-nexus-via`** — **Decided: natural behavior, no
special handling.**

When CP rejects via hook and AI Gateway is never invoked, `x-nexus-via:
compliance-proxy` (no `ai-gateway` segment). Same when Agent rejects upstream
of CP. This is exactly the "every hop that processed prepends itself" rule
applied — `via` reflects what was *actually* on the path, which is the most
honest signal for an operator. No code action.

**D-3 — `x-nexus-{cp,agent}-domain-rule` value** — **Decided: emit the rule
UUID, not the rule name.**

Interception-domains rule names are admin-defined strings and may leak internal
policy naming if exposed. UUID keeps the wire surface stable and admin-only;
operators map UUID → name via Admin API as needed. Consistent with the
project's IoT-terminology boundary (internal labels stay off the wire).

**D-4 — Inter-service hook dedup** — **Decided: out of scope for this spec;
two follow-up specs committed.**

This spec delivers the **producer** half (every service declares its hook
outcomes in response markers). The **consumer** half is split into two
follow-up specs:

- **Next spec (P1 — Agent response-side hooks + response-marker dedup):**
  starts immediately after Phase 0 of this spec lands. Scope: (a) extend the
  Agent to execute response-side hooks (a capability it lacks today; not a
  cosmetic toggle — needs design for hook scoping, end-user-side latency
  budget, and transform-conflict resolution between Agent and CP), (b) wire
  dedup so the Agent skips any hook already listed in incoming
  `x-nexus-cp-hook` / `x-nexus-aigw-hook`. P1 is small *because* Phase 0 of
  this spec pays the response-handling cost upfront; it does **not** start
  before Phase 0 is merged.
- **Later spec (P2 — Request-side dedup):** mirror marker on the request path
  (e.g. `x-nexus-request-processed: pii-redact,jwt-strip`) so downstream
  services can skip already-executed hooks at request time. Requires hook-
  version negotiation across services and a trust boundary (the marker must
  come from another Nexus service, not a forged client header). No date
  commitment; filed as a tracked future spec.

The asymmetry to remember: Agent is the **first** hop on the request path so
it cannot detect downstream-already-processed at request time; it is the
**last** hop on the response path so it can dedup naturally on responses. P1
exploits this asymmetry on the response side; P2 needs extra machinery to do
the same on the request side.

**Current-spec action:** none for D-4. The producer markers landed by this
spec are sufficient for P1 to consume in the next spec.

## 13. Out of Scope (explicitly deferred)

- Response-side hooks in Agent (Phase 0 unblocks this; the actual hooks land
  later under their own SDD).
- Marker visibility for CONNECT-tunnel / non-MITM flows. Such flows are
  observable only via the Nexus Hub flow inventory and audit log.
- Customer-facing stable API contract for the markers. The contract is
  developer-/operator-facing during pre-GA; long-term stability is revisited at
  GA.
- Inter-service hook dedup (see D-4). Response-side dedup (P1) starts as the
  next spec after Phase 0; request-side dedup (P2) is a tracked future spec.

## 14. Acceptance Summary (for the implementation plan)

The plan that follows this design must produce, in order:
1. **Phase 0** — `packages/shared/transport/responseio` (or chosen name) extracted from
   compliance-proxy; CP migrated; CP tests still green.
2. **Phase 1** — Shared helpers (`PrependVia`, `MergeExposeHeaders`,
   `SetExposeHeaders`, `ExposeHeaders` slice, `formatHookOutcome`) in
   `packages/shared/traffic`, with unit tests.
3. **Phase 2** — AI Gateway rename + new `aigw-hook` field + `aigw-coerced`
   transparency header (D-1) + CORS expose; unit and integration tests updated.
4. **Phase 3** — Compliance-Proxy markers (with `cp-domain-rule` as UUID per
   D-3) + reject-path migration + CORS merge; tests added.
5. **Phase 4** — Agent markers via Phase 0 infra (with `agent-domain-rule` as
   UUID per D-3) + reject-path migration + CORS merge; tests added including
   the negative tunnel-flow test.
6. **Phase 5** — Cross-service e2e and documentation deliverables (§12).
